### hw1_1.py — 잠재적 친구 추천 (Spark)

**핵심 아이디어**: 어떤 사용자 u의 친구 목록이 [f1, f2, f3, ...]이면, f1과 f2는 u를 통해 공통 친구를 하나 가진다. 따라서 모든 사용자의 친구 목록에서 친구 쌍을 생성하고, 각 쌍이 몇 번 등장하는지 세면 공통 친구 수가 된다.

#### 전체 흐름

```
입력 파일 (인접 리스트)
    ↓ parse_line
(user, [friends]) RDD
    ↓ .cache()  ← 두 번 사용하므로 캐싱
    ├──→ emit_existing_friendships → ((min,max), True) 직접 친구 쌍
    └──→ emit_mutual_candidates   → ((min,max), 1)    후보 쌍
                                        ↓ reduceByKey(+)
                                  공통 친구 수 RDD
                                        ↓ subtractByKey(직접 친구 쌍)
                                  직접 친구를 제거한 RDD
                                        ↓ sortBy(-count, u1, u2).take(10)
                                  상위 10쌍 출력
```

#### 함수별 설명

**`parse_line(line)`**  
탭으로 구분된 줄을 파싱해 `(사용자ID, [친구ID 리스트])`를 반환한다.  
친구가 없는 사용자(탭 이후가 비어있는 경우)도 처리한다.

**`emit_mutual_candidates(record)`**  
사용자 u의 친구 목록 F에서 모든 쌍 (fi, fj)을 생성하고,  
각 쌍에 대해 `((min(fi,fj), max(fi,fj)), 1)`을 방출한다.  
→ 나중에 `reduceByKey(+)`로 합산하면 공통 친구 수가 된다.  
→ `(min, max)`로 정규화해서 (A,B)와 (B,A)를 중복 없이 하나로 처리한다.

**`emit_existing_friendships(record)`**  
사용자 u와 각 친구 f에 대해 `((min(u,f), max(u,f)), True)`를 방출한다.  
→ 이 키 집합을 이용해 실제 친구 쌍을 제거(마스킹)한다.

**`subtractByKey`를 선택한 이유**  
직접 친구 집합이 매우 클 수 있으므로, 브로드캐스트 방식 대신 Spark의 셔플 기반 조인인 `subtractByKey`를 사용한다. 키 일치 여부로 제거하는 깔끔한 방법이다.

**정렬 기준**:  
`sortBy(lambda x: (-x[1], x[0][0], x[0][1]))` — 공통 친구 수 내림차순, 첫 ID 오름차순, 두 번째 ID 오름차순.

### hw1_2_b.py — A-Priori 알고리즘 (빈발 쌍 + 연관 규칙)

**핵심 아이디어**: A-Priori는 2단계 패스로 동작한다. 1패스에서 빈발 단일 아이템을 찾고, 2패스에서 빈발 아이템끼리의 쌍만 세어 메모리를 절약한다.

#### 전체 흐름

```
입력 파일 (browsing.txt)
    ↓ Pass 1: 모든 아이템 카운트 + 전체 바스켓 메모리 로드
빈발 아이템 집합 (지지도 >= 100)
    ↓ 사전순 정렬 → 인덱스 부여 (0..N-1)
    ↓ Pass 2: 삼각 행렬로 빈발 아이템 쌍 카운트
빈발 쌍 목록 (지지도 >= 100)
    ↓ 연관 규칙 생성 (신뢰도 >= 50%)
출력 (12줄)
```

#### 삼각 행렬(Triangular Matrix) 설명

N개의 빈발 아이템에 대해 모든 쌍 (i, j) (i < j)의 카운트를 1차원 배열에 저장한다.

인덱스 공식:
```
index(i, j) = i*N - i*(i+1)//2 + (j - i - 1)
```

예: N=4이면 쌍은 (0,1), (0,2), (0,3), (1,2), (1,3), (2,3) 총 6개.
- `index(0,1)=0`, `index(0,2)=1`, `index(0,3)=2`
- `index(1,2)=3`, `index(1,3)=4`
- `index(2,3)=5`

배열 크기: `N*(N-1)//2` — (i,j)와 (j,i)를 중복 저장하지 않아 절반으로 절약.  
`numpy int32` 배열을 사용해 Python 리스트보다 메모리를 훨씬 적게 사용한다.

#### 주요 설계 선택

- **Pass 1에서 전체 바스켓을 메모리에 로드**: 파일 크기가 3.4 MB로 작아서 가능. Pass 2에서 파일을 다시 읽지 않아도 된다.
- **빈발 아이템을 사전순 정렬**: 동일한 실행마다 같은 인덱스 매핑을 보장 (재현성).
- **신뢰도 계산**: `Confidence(A→B) = support({A,B}) / item_counts[A]`  
  각 빈발 쌍 (A, B)에 대해 A→B와 B→A 두 방향 모두 검사한다.
- **정렬**: 신뢰도 내림차순, 동점 시 지지도 내림차순.

### hw1_3_b.py — MinHash 기반 LSH (문서 유사도)

**핵심 아이디어**: 문서를 Jaccard 유사도가 높은 것끼리 찾기 위해, 문서를 고차원 shingle 집합으로 표현하고 MinHash로 압축한 후, LSH 밴드 기법으로 유사한 것만 비교한다.

#### 전체 파이프라인

```
articles.txt
    ↓ Step 1: 기사 파싱 (ID, 텍스트)
    ↓ Step 2: 3-shingle 추출 (알파벳+공백, 소문자)
    ↓ Step 3: 전역 shingle 목록 사전순 정렬 → 행 인덱스
    ↓ Step 4: MinHash 서명 행렬 계산 (D x 120)
    ↓ Step 5: LSH 밴드 버킷 구성
    ↓ Step 6: 후보 쌍의 서명 유사도 계산 → 0.9 이상 출력
```

#### 3-shingle 추출

```python
def extract_shingles(text, k=3):
    cleaned = [ch for ch in text.lower() if ch.isalpha() or ch == ' ']
    s = ''.join(cleaned)
    return {s[i:i+k] for i in range(len(s)-k+1)}
```
- 소문자 변환 후 알파벳과 공백만 남김
- 3글자 슬라이딩 윈도우로 shingle 집합 생성 (set → 자동 중복 제거)

#### MinHash 서명 계산 (벡터화)

해시 함수 형태: `h(x) = (a*x + b) % c`  
- c = shingle 수(n) 이상의 최소 소수
- a, b는 [0, c-1]에서 균일 랜덤 (`np.random.seed(0)`)
- 총 120개의 해시 함수 (b=6 밴드 × r=20 행)

```python
# rows: 이 문서의 shingle 행 인덱스들, shape: (|shingles|,)
hash_mat = (a_arr[:, None] * rows[None, :] + b_arr[:, None]) % c
# shape: (120, |shingles|) — 한 번의 행렬 연산으로 모든 해시값 계산
sig[doc_idx] = hash_mat.min(axis=1)  # 각 해시 함수의 최솟값
```
Python 루프 없이 numpy 행렬 연산으로 120개 해시를 한꺼번에 계산 → 빠름.

#### LSH 밴드 기법

- 서명(120개)을 6개 밴드(각 20개 값)로 나눔
- 같은 밴드에서 동일한 20개 값을 가진 문서들을 같은 버킷에 묶음
- 하나 이상의 밴드에서 같은 버킷에 속하면 후보 쌍으로 등록

이론적 임계값: `t ≈ (1/b)^(1/r) = (1/6)^(1/20) ≈ 0.9`

#### 서명 유사도

```python
similarity = float(np.sum(sig[i] == sig[j])) / num_hashes
```
120개 해시 위치 중 일치하는 비율 = Jaccard 유사도의 추정치.

### hw1_3_c.py — TF-IDF + 랜덤 초평면 LSH (코사인 유사도)

**핵심 아이디어**: 문서를 TF-IDF 벡터로 표현하고, SimHash(랜덤 초평면)로 유사한 문서를 같은 버킷에 묶은 후, 후보 쌍에 대해 정확한 코사인 유사도를 계산한다.

#### 전체 파이프라인

```
articles.txt
    ↓ Step 1: 파싱
    ↓ Step 2: 전역 어휘 사전순 정렬 → 차원 인덱스
    ↓ Step 3: TF-IDF 벡터 계산 (희소 딕셔너리)
    ↓ Step 4: 랜덤 초평면 10개로 LSH 버킷 구성
    ↓ Step 5: 버킷 내 후보 쌍 코사인 유사도 계산
    ↓ 코사인 거리 < 0.1인 쌍 출력
```

#### TF-IDF 계산

- **TF(t, d)**: 문서 d에서 단어 t의 등장 횟수 (raw count)
- **IDF(t)**: `log(N / df(t))` (자연 로그, N = 전체 문서 수)
- **TF-IDF**: `TF * IDF`

**희소 딕셔너리 사용 이유**: 전체 TF-IDF 행렬(2000개 문서 × ~25,000 단어)을 dense 배열로 만들면 ~400 MB가 필요. 대부분의 문서는 전체 어휘의 극히 일부만 포함하므로, `{단어인덱스: tfidf값}` 딕셔너리로 저장해 메모리를 크게 절약한다.

#### 랜덤 초평면 LSH (SimHash)

```python
hyperplanes = np.random.randn(10, V)  # 10개의 랜덤 초평면
projections[doc_idx] += tfidf[t_idx] * hyperplanes[:, t_idx]  # 항 단위 누적
hashes = (projections > 0)  # 10비트 버킷 키
```

- 10개의 초평면 각각에 대해, 문서 벡터가 초평면의 양쪽 중 어디에 있는지를 나타내는 1비트를 생성
- 10비트 키 → 최대 1024개 버킷
- **항 단위 누적 방식**: 전체 D×V TF-IDF 행렬을 메모리에 올리지 않고, 어휘 하나씩 처리하며 10차원 내적을 누적

#### 코사인 유사도 계산

```python
dot = sum(v * db[k] for k, v in da.items() if k in db)
cos_sim = dot / (norms[a] * norms[b])
cos_dist = 1.0 - cos_sim
```

- 두 희소 딕셔너리의 교집합 키에 대해서만 내적 계산 → 효율적
- 더 짧은 딕셔너리를 기준으로 루프 → 추가 최적화
- 코사인 거리 = 1 - 코사인 유사도; 0.1 미만인 쌍만 출력 (유사도 > 0.9)

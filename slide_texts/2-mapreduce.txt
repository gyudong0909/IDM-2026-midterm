Physical Organization of Nodes
• Parallel-computing architecture:
  • Compute nodes are stored on racks (perhaps 8 – 64 on a rack).
  • The nodes on a single rack are typically connected by gigabit Ethernet.
  • There can be many racks connected by a switch.

Machine Failures
• Machines fail frequently:
   • One server may stay up 3 years (≈ 1,000 days).
   • If you have 1,000 servers, expect to lose 1/day.
   • With 1M machines, 1,000 machines fail every day!
• Q: If nodes fail, how to store data persistently?
   • Loss of a single node (e.g., disk crashes)
   • Loss of an entire rack (e.g., network fails)

Storage Infrastructure
• Distributed File System (DFS) is a solution.
   • Hadoop Distributed File System (HDFS) or Google File System (GFS).
• Typical usage pattern:
   • Huge files (100s of GB to TB).
   • Reads and appends are common.
   • Data is rarely updated in place.




                                   Jaemin Yoo                             7
Distributed File System
• Chunk nodes (or data nodes):
  • Store data divided into chunks, which are typically 16 – 64 MB.
  • Chunks are replicated (usually 2x or 3x) at different nodes.
     • Try to keep replicas in different racks.
• Master nodes (or head/name nodes):
  • Runs a master process that manages the entire file system.
  • Stores the metadata (i.e., master file) about where files are stored.
  • Typically has a backup, which takes over when the node fails.

Distributed Computation
• Q: How do you distribute computation?
  • Copying large data over a network takes time.
  • We’ll learn MapReduce and Spark as solutions.
• Idea: Bring computation to data!
  • Make chunk servers also serve as compute servers.
  • Run a (distributed) algorithm for each chunk.
  • Gather the result, typically smaller than the raw data.

MapReduce
• MapReduce is a style of programming designed for
   1. Easy parallel programming
   2. Invisible management of hardware and software failures
   3. Easy management of very-large-scale data
• It has several implementations, including Hadoop, Flink, etc.
• The original Google implementation is called “MapReduce”.

Example: Word Counting
• Suppose we have a huge text document.
• Goal: Count the number of times each distinct word appears:
  • E.g., (ball, 30), (basket, 10), (great, 5), and so on.
• Many applications:
  • Analyze web server logs to find popular URLs.
  • Analyze market data to find popular items.




                                       Jaemin Yoo               13
MapReduce Components
• Map:
  • Apply a user-written Map function to each input element.
  • The output of the Map function is a set of key-value pairs.
• Group by key:
  • System organizes all the key-value pairs by key.
  • Return key-“value list” pairs.
• Reduce:
  • Apply a user-written Reduce function to each key-“value list” pair.
  • The output of the Reduce function is a key-value pair.

Map Task
• Input:
   • Chunks consisting of elements, which are key-value pairs.
      • Why pairs? To allow composition of several MapReduce processes.
      • Keys of the input (i.e., initial) elements can be ignored.
• Output:
   • Zero or more key-value pairs.

Group by Key
• Input:
   • Key-value pairs from Map tasks.
• Output:
   • The master controller knows the number 𝑟 of Reduce tasks.
   • Apply a hash function to the keys, producing bucket numbers 0 to 𝑟 − 1.
   • The key-value pairs from each Map task are put into 𝑟 local files:
      • Each local file is destined for a specific Reduce task.
   • Merge the files from multiple Map tasks and return 𝑘, 𝑣!, 𝑣", … , 𝑣# .




                                           Jaemin Yoo                          17
Reduce Task
• Input:
   • One or more keys and their associated value lists.
      • One Reducer can take several keys.
• Output:
   • The Reduce function is applied to each key-“value list” pair.
   • The outputs from all the Reduce tasks are merged into a single file.

MapReduce for Word Counting
// key: line index; value: each line of the document
map(key, value):
     for each word w in value:
           emit(w, 1)

// key: a word; values: an iterator over counts
reduce(key, values):
     result = 0
     for each count v in values:
           result += v
     emit(key, result)

Combiners
• Suppose the Reduce function is commutative and associative.
  • Commutative: 𝑎 + 𝑏 = 𝑏 + 𝑎.
  • Associative: 𝑎 + 𝑏 + 𝑐 = 𝑎 + 𝑏 + 𝑐 .
  • E.g., word counting, matrix-vector multiplication.
• Then, values can be combined in any order.
• → The Reducer’s work can be pushed to the Map tasks.
  • E.g., instead of producing (𝑤, 1), (𝑤, 1), …, produce (𝑤, 𝑚).
  • The amount of data shuffling for Group By Key is greatly reduced.

MapReduce Program Execution
• User program forks a Master controller and Worker processes.
  • A worker handles either Map or Reduce tasks, but not both.




                                                                 25
Master Controller
• Master controller:
  • Schedules Map and Reduce tasks and assigns them to Workers.
     • One Map task per chunk is reasonable.
     • Reduce tasks should be fewer than Map tasks.
         • Otherwise, too many intermediate files per Map task.
         • Will be discussed later in detail.
  • Keeps track of task status (idle, executing at a Worker, or completed).
  • Worker process reports to Master when it finishes a task.




                                           Jaemin Yoo                         26
Map and Reduce Tasks
• Map task:
  • Creates a file for each Reduce task on the Map Worker’s local disk.
  • The Master is informed of the files and the destination Reduce tasks.
• Reduce task:
  • When assigned to a Worker, it is given all the files that form its input.
  • Executes code written by the user and writes its output to a file.




                                     Jaemin Yoo                                 27
MapReduce Pipelining
• Reduce workers start reading data as soon as a Map task finishes.
• Reduce tasks start after reading all data.

Coping with Node Failures
• Master failure:
   • Entire MapReduce job must be restarted.
• Map worker failure:
   • Detected by Master because it periodically pings Worker processes.
   • All Map tasks assigned this worker have to be redone, even if completed.
      • Output for Reduce tasks reside in compute node and are unavailable.
   • Master informs each Reduce task that input location has changed.
• Reduce worker failure:
   • Only in-progress tasks are reset to idle and the Reduce task is restarted.
      • Since the output is stored in DFS, not a local disk.

                                           Jaemin Yoo                             29
Number of Reduce Tasks
• Pipelining can be optimized by using one Reduce task per key.
• However, this plan is not usually the best.
   • Overhead associated with each task is created.
   • Often there are far more keys than there are compute nodes.
   • Significant variation in the # of values for different keys.
      • Reduce tasks will exhibit skew in processing time.
• Rule of thumb: # of nodes < # of Reduce tasks < # of keys.

Algorithms using MapReduce
• MapReduce is not a solution to every problem.
  • DFS makes sense only when files are large and rarely updated in place.
  • MapReduce is not suitable for frequent database changes.
     • E.g., managing online retails sales (say by Amazon).
• Original purpose of Google:
  • To execute large matrix-vector multiplications (for PageRank).
  • Matrix-vector calculations fit nicely into MapReduce.
  • Relational algebra (i.e., SQL) can also use MapReduce effectively.

• What if the vector 𝐯 cannot fit in main memory?
   • Divide 𝑀 into vertical stripes and divide 𝐯 into horizontal stripes.
      • The width of vertical strips = the height of horizontal stripes.
      • Use enough stripes so that one stripe of 𝐯 fits in main memory.
   • Each Map task is assigned a chunk from corresponding stripes.
   • The Map and Reduce tasks then run exactly the same as before.

L1: Introduction to Data Mining

Intuition

Why Data Mining?

"Data creates value only when it can be turned into decisions. The amount of data itself does not mean anything." - Prof. Yoo
"Data mining is not about memorizing the specific details. You just have to be sure about the intuition and the core assumption." - Prof. Yoo

Data Mining = process of transforming raw data into insights. Three stages: (1) Store data, (2) Manage data, (3) Analyze data.

Data Mining ≈ Data Science ≈ Data-Centric AI

Theory

Key Concepts

TF-IDF (Term Frequency - Inverse Document Frequency)

\[\text{TF-IDF}(i, j) = \text{TF}(i, j) \times \text{IDF}(i)\] \[\text{TF}(i, j) \propto \text{# occurrences of term } i \text{ in document } j\] \[\text{IDF}(i) \propto \frac{1}{\text{# documents where term } i \text{ appears}}\]

Intuition: High TF-IDF = term appears often in THIS document but rarely across ALL documents. Words like "the" have high TF but low IDF. Rare jargon has low TF but high IDF. Topical keywords have both high.

Bonferroni's Principle

When looking for patterns in data, compare with random data of the same size. If the expected number of occurrences in random data is large, the pattern is likely meaningless (bogus).

Birthday Problem: With 23 people, P(shared birthday) > 0.5. Seems surprising but is statistically expected.

Hash Functions

A hash function \(h\) maps a key to a bucket number \(\in [0, B-1]\). Enables O(1) data structure for big data. Collisions are inevitable when \(B < \#\text{unique keys}\).

Power Laws

\[\log y = a \log x + b \quad \Leftrightarrow \quad y = c \cdot x^a\]

Linear in log-log scale. Examples: sales of products, sizes of websites, wealth of people (80/20 rule).

Memory Hierarchy

L0 Registers > L1 Cache (SRAM) > L2/L3 Cache > L3 Main Memory (DRAM) > L4 Local Storage (SSD/HDD) > L5 Remote/Cloud Storage (DFS). Higher = faster but smaller/costlier. Algorithm design depends on where data is stored.

Example

When is ML Useful?

  • ML useful: When we know WHAT we want but not HOW (recommendation, forecasting, embedding)
  • ML not necessary: When we can describe the goal directly (locating resumes on the web, parsing HTML)
Exam Points

Must-Know for L1

  • TF-IDF formula and meaning of each component
  • Bonferroni's principle: why patterns in random data are meaningless
  • Hash function: key -> bucket, collisions inevitable
  • Power law: log-log linear relationship
  • Memory hierarchy: what bottleneck changes at each level
  • T/F Trap: "Data mining is all about machine learning" -> FALSE (it's about using ML as a tool)

L2: MapReduce

Intuition

Why Distributed Computing?

"We need to bring computation to data. These chunk servers store the data, we distribute our algorithm to all these chunk servers, so we don't exchange data." - Prof. Yoo
"MapReduce is basically a style of programming. It is not a solution to every problem." - Prof. Yoo

Key insight: 1000 servers -> expect 1 failure/day. With 1M machines, 1000 fail daily. Need systematic fault tolerance.

Theory

Distributed File System (DFS)

  • Chunk nodes (data nodes): Store data in chunks (16-64 MB), replicated 2-3x at different nodes (different racks)
  • Master node (name node): Manages metadata, has backup
  • Usage pattern: Huge files, reads/appends common, data rarely updated in place

MapReduce Components

\[\text{Map}: (k, v) \to \{(k_i, v_i)\}_i\] \[\text{Group by Key}: \{(k, v)\} \to (k, [v_1, v_2, \ldots, v_n])\] \[\text{Reduce}: (k, [v_1, \ldots, v_n]) \to (k, v)\]

Three Steps:

  1. Map: User-written function applied to each input element. Output: set of key-value pairs. Fully parallelizable.
  2. Group by Key: System gathers all pairs with same key (conceptual step, done via hash-based shuffling).
  3. Reduce: User-written function applied to each key-value list pair. Aggregates values.

Combiner Optimization

If Reduce function is commutative (\(a+b = b+a\)) and associative (\((a+b)+c = a+(b+c)\)), then values can be combined in any order. Reducer's work can be pushed to Map tasks, producing \((w, m)\) instead of \((w,1),(w,1),\ldots\). Greatly reduces shuffling.

Node Failure Handling

Failure TypeActionReason
Master failureEntire job restartsToo expensive to replicate master state
Map worker failureALL tasks (even completed) must redoMap outputs on local disk, inaccessible after failure
Reduce worker failureOnly in-progress tasks restartCompleted reduce outputs stored in DFS (replicated)

Matrix-Vector Multiplication in MapReduce

Case 1: v fits in memory

// Map: input key=(i,j), value=m_ij
map((i,j), m_ij):
    emit(i, m_ij * v[j])     // v is available in memory

// Reduce: sum all values for same row i
reduce(i, values):
    emit(i, sum(values))

Case 2: v does not fit in memory -- Divide M into vertical stripes, v into horizontal stripes. Each Map task takes one stripe of M and corresponding stripe of v.

Number of Reduce Tasks

Rule of thumb: # nodes < # Reduce tasks < # keys. One Reduce task per key is theoretically max parallel but impractical.

Example

Word Count - Complete Flow

// Map function
map(key, value):                // key: line index, value: line text
    for each word w in value:
        emit(w, 1)

// Reduce function
reduce(key, values):           // key: word, values: [1, 1, ..., 1]
    result = 0
    for each count v in values:
        result += v
    emit(key, result)

With Combiner: Each Map task locally combines (w,1),(w,1),(w,1) -> (w,3) before shuffling.

Mutual Friends (Pop Quiz from Lecture)

// Input: A -> B, C, D (A is friends with B, C, D)
map(key, friends):             // key: person, friends: friend list
    for each pair (w, v) in friends:
        emit((w,v), key)       // pair of friends -> original person

// Reduce: identity (just collect)
reduce((w,v), values):
    emit((w,v), values)        // output mutual friends list
Exam Points

Must-Memorize for L2

  • Map/Reduce function signatures: Map: (k,v) -> {(k_i,v_i)}, Reduce: (k,[v1,...,vn]) -> (k,v)
  • Combiner condition: Reduce must be commutative AND associative
  • Why Map failure redoes ALL tasks: outputs on LOCAL disk
  • Why Reduce failure redoes only in-progress: outputs on DFS
  • Group by Key is conceptual: no explicit GBK worker exists
  • MV multiplication map function: ((i,j), m_ij) -> (i, m_ij * v_j)
T/F Traps:
- "Programmer writes GroupByKey function" -> FALSE (system handles it)
- "Framework checks commutativity/associativity" -> FALSE (programmer's responsibility)
- "MapReduce is suitable for ML training" -> FALSE (not good for frequent data updates)

L3: Spark

Intuition

Why Spark over MapReduce?

"MapReduce creates a chain of tasks. Let's extend it into a directed acyclic graph (DAG) of tasks." - Prof. Yoo
"Spark is more flexible framework compared to MapReduce. It is much easier for a programmer to write Spark." - Prof. Yoo

MapReduce limitations: (1) Stores intermediate files on disk always (2) Substantial overhead for complex/iterative jobs due to HDFS writes between iterations (3) Many algorithms hard to express as Map+Reduce

Theory

Spark Core Concepts

Hadoop Stack

Database (HDFS) -> Resource Management (YARN) -> Processing Engine (MapReduce/Spark) -> Libraries (SparkSQL, MLlib, GraphX)

RDD (Resilient Distributed Dataset)

  • Partitioned collection of records of one type (generalizes MapReduce key-value pairs)
  • Spread across cluster (chunks), read-only (immutable)
  • Can be cached in memory
  • Created from DFS or by transforming other RDDs

Operations: Transformations vs Actions

Transformations (RDD -> RDD)Actions (RDD -> Value)
map, flatMap, filter, join, union, distinct, groupByKey, reduceByKey, sortByKeycount, collect, reduce, save, take
Lazy: not executed until actionForces computation

Key Operations Detail

OperationTypeDescriptionDependency
flatMapTransformation1 element -> 0 or more elements, flattenedNarrow
mapTransformation1 element -> exactly 1 elementNarrow
filterTransformationKeep elements satisfying predicateNarrow
groupByKeyTransformationGroups values by key (shuffle)Wide
reduceByKeyTransformationgroupByKey + reduce (optimized)Wide
joinTransformationMerges two RDDs on keyWide
reduceActionAggregates all elements to single value-
collectActionReturns RDD to local program-

Wide vs Narrow Dependency

  • Narrow: Each partition depends on at most one parent partition (parallelizable, fast)
  • Wide: Each partition depends on multiple parent partitions (requires shuffling, slow)
  • flatMap is Narrow: Even though it can create multiple partitions from one, each output partition still comes from only one input partition

Blocking Property (DAG Fault Tolerance)

Operation runs only after all predecessors complete. If task fails, no successor started -> master controller knows exactly what to restart.

Two Key Improvements over MapReduce

  1. Lazy Evaluation: Spark doesn't apply transformations until an action is called. Allows optimization of execution plans, combining transformations, in-memory passing.
  2. Lineage: Each RDD tracks its parent RDD. If a chunk is lost, trace back through lineage to reconstruct from predecessor. Lineage is at chunk level.

MapReduce vs Spark Comparison

MapReduceSpark
SpeedFaster than traditionalUp to 100x faster
LanguageJavaScala (Python API available)
ProcessingBatch onlyBatch/real-time/iterative/interactive
CachingNoneIn-memory caching
Ease of UseComplex, verboseCompact, easier
Best ForLinear processing, simple batchInteractive, iterative (ML), real-time
Example

Average Word Length by Letter (Spark)

avglens = sc.textFile(file) \
    .flatMap(lambda line: line.split()) \        # lines -> words (narrow)
    .map(lambda word: (word[0], len(word))) \     # word -> (first_letter, length) (narrow)
    .groupByKey() \                               # group by letter (WIDE - shuffle!)
    .map(lambda (k, values): \                    # compute average (narrow)
        (k, sum(values) / len(values)))

Word Count using reduceByKey (Pop Quiz)

counts = sc.textFile(file) \
    .flatMap(lambda line: line.split()) \         # split lines into words
    .map(lambda w: (w, 1)) \                     # each word -> (word, 1)
    .reduceByKey(lambda a, b: a + b)             # sum counts per key

Key insight from lecture: Reduce in MapReduce = reduceByKey in Spark. Map in MapReduce = flatMap in Spark. They have different names!

Exam Points

Must-Memorize for L3

  • RDD = partitioned, immutable, distributed collection of one type
  • Transformation vs Action distinction; Lazy evaluation meaning
  • Wide dependency: groupByKey, join, sort. Narrow: map, flatMap, filter
  • flatMap is NARROW even though it creates variable output sizes
  • Lineage for fault recovery (trace back to reconstruct lost RDD)
  • reduceByKey is better than groupByKey + reduce (gives system more optimization room)
T/F Traps:
- "RDDs can be updated in place" -> FALSE (immutable)
- "Spark always outperforms MapReduce" -> FALSE (not when data exceeds memory)
- "flatMap creates wide dependency" -> FALSE (narrow)
- "sort creates wide dependency" -> TRUE

L4: Frequent Itemsets 1

Intuition

Why Frequent Itemsets?

"In a supermarket, the manager wants to find patterns of items. If someone buys diaper and milk, he is likely to buy beer. Don't be surprised if you find six-packs next to diapers!" - Prof. Yoo
"Market-basket model is an abstract of various data. Items do not have to be 'in' baskets!" - Prof. Yoo

Core problem: Finding frequent itemsets is HARDER than finding association rules. Once we have frequent itemsets, rules are easy to generate.

Why pairs are hardest: As itemset size grows, probability of being frequent drops exponentially. Frequent pairs are common, triples are rare.

Theory

Market-Basket Model

  • Items: Set of all things (e.g., products in store)
  • Baskets: Subsets of items (e.g., one customer's purchase)
  • Abstract: Any many-to-many mapping. Topic discovery (items=words, baskets=docs), plagiarism (items=docs, baskets=sentences), biomarkers (items=drugs, baskets=patients)

Key Definitions

\[\text{Support}(I) = \text{# baskets containing all items in } I\] \[\text{Itemset } I \text{ is frequent if } \text{Support}(I) \geq s\]

Association Rules: \(I \to J\) where \(I \cap J = \emptyset\)

\[\text{Support}(I \to J) = \text{Support}(I \cup J)\] \[\text{Confidence}(I \to J) = \frac{\text{Support}(I \cup J)}{\text{Support}(I)} = P(J|I)\] \[\text{Interest}(I \to J) = \text{Confidence}(I \to J) - P(J) = P(J|I) - P(J)\]

Interest interpretation: Close to 0 = rule is not interesting (J appears frequently regardless of I). Positive = positive correlation. Negative = negative correlation. Goal: find rules with support >= s AND confidence >= c.

Mining Association Rules (Two Steps)

  1. Step 1: Find all frequent itemsets I (the hard part)
  2. Step 2: For every subset \(A \subset I\), generate rule \(A \to I \setminus A\). Check confidence. Confidence can only decrease when moving items from LHS to RHS (because numerator stays same, denominator can only decrease). Use this to prune.

Closed vs Maximal Frequent Itemsets

ConceptDefinitionProsCons
ClosedNo immediate superset has the same supportSafe compaction, preserves support infoStill numerous
MaximalNo immediate superset is frequentMore aggressive pruningLoses support details of subsets

All maximal are closed, but not all closed are maximal.

Counting Pairs in Memory

MethodMemoryWhen Better
Triangular Matrix\(4 \times \frac{n(n-1)}{2}\) bytesDense data (p > 1/3)
Triples (i, j, count)\(12 \times p \times \frac{n(n-1)}{2}\) bytesSparse data (p < 1/3)

A-Priori Algorithm

Key idea: Monotonicity -- If an itemset I appears >= s times, so does every subset J of I. Contrapositive: if item i is infrequent, no pair containing i can be frequent.

Two-Pass Algorithm:

  1. Pass 1: Read baskets, count occurrences of each item. Memory: O(n). Mark items with count >= s as frequent.
  2. Between passes: Create new numbering 1..m for frequent items. Non-frequent items get index 0.
  3. Pass 2: Read baskets again. For each basket, look at only frequent items. Count all pairs of frequent items. Memory: O(m^2).

Generalization Pipeline:

\[C_1 \xrightarrow{\text{Filter}} L_1 \xrightarrow{\text{Construct}} C_2 \xrightarrow{\text{Filter}} L_2 \xrightarrow{\text{Construct}} C_3 \xrightarrow{\text{Filter}} L_3 \to \cdots\]

\(C_k\) = candidate k-tuples, \(L_k\) = truly frequent k-tuples. Each Filter requires a disk pass. Each Construct does not.

Example

Complete A-Priori Example

Items = {milk, coke, pepsi, beer, juice}, s = 3, 8 baskets:

B1={m,c,b}, B2={m,p,j}, B3={m,b}, B4={c,j}, B5={m,p,b}, B6={m,c,b,j}, B7={c,b,j}, B8={b,c}

Pass 1: Count items: m=5, c=5, p=2, b=6, j=4. Frequent (>=3): m, c, b, j. Pepsi eliminated.

Pass 2: Count pairs of {m,c,b,j} only:

Pair{m,c}{m,b}{m,j}{c,b}{c,j}{b,j}
Count342432

Frequent pairs (>=3): {m,c}, {m,b}, {c,b}, {c,j}

Support/Confidence/Interest Example

Rule: {m,b} -> c. Support({m,b,c}) = 2. Support({m,b}) = 4.

Confidence = 2/4 = 0.5. P(c) = 5/8. Interest = 0.5 - 5/8 = -1/8 (negative!).

Exam Points

Must-Memorize for L4

  • Formulas: Support, Confidence, Interest definitions
  • A-Priori monotonicity: If I is frequent, all subsets of I are frequent
  • Contrapositive: If item i infrequent, no superset containing i is frequent
  • Triangular vs Triples: Triples better when p < 1/3
  • Closed vs Maximal: All maximal are closed. Maximal loses support info.
  • Finding rules is easy AFTER finding frequent itemsets
  • Cost measured by # disk passes, not algorithmic complexity
T/F Traps:
- "A-Priori finds interesting rules" -> FALSE (finds frequent itemsets, uses support not interest)
- "Interest is always positive" -> FALSE (can be negative)
- "Market-basket applies only to retail" -> FALSE (abstract model)
- "All closed itemsets are maximal" -> FALSE (reverse is true)

L5: Frequent Itemsets 2

Intuition

Why Improve A-Priori?

"The A-Priori algorithm is fine as long as counting candidate pairs can be done in main memory. What if the memory is not enough?" - Prof. Yoo
"In many data mining algorithms, we care about the trade-off between accuracy or approximation and speed, unlike a theoretical algorithm class." - Prof. Yoo

Problem: Even after A-Priori's Pass 1 filtering, candidate pairs C_2 may still exceed memory. Need algorithms to cut down |C_2| further.

Theory

1. PCY (Park-Chen-Yu) Algorithm

Motivation: In Pass 1 of A-Priori, most memory is idle (only storing item counts). Use idle memory for a hash table of pairs.

How it works:

  1. Pass 1: Count items (same as A-Priori) AND hash every pair {i,j} to a bucket, incrementing bucket count.
  2. Between passes: Replace hash table with bit-vector (bitmap). 1 = bucket count >= s ("frequent bucket"), 0 = not. Reduces space by 1/32 (integer -> 1 bit).
  3. Pass 2: Candidate pair {i,j} must satisfy TWO conditions:
    • (1) Both i and j are frequent items
    • (2) {i,j} hashes to a frequent bucket in bitmap

PCY must use triples method (not triangular matrix) because it exploits sparsity of candidate pairs.

2. Multistage Algorithm

Idea: Create successive hash tables with different hash functions in multiple passes.

  1. Pass 1: Same as PCY -> produces Bitmap 1
  2. Pass 2: Using Bitmap 1 + frequent items, create second hash table (only hash pairs satisfying conditions 1+2). Second table has 31/32 of original buckets (due to Bitmap 1 memory). -> produces Bitmap 2
  3. Pass 3: Count pairs satisfying ALL three conditions:
    • (1) Both items frequent
    • (2) Hash to frequent bucket in Bitmap 1
    • (3) Hash to frequent bucket in Bitmap 2
Critical subtlety (exam favorite): Condition 2 (Bitmap 1) CANNOT be dropped even though we have Bitmap 2! Because a pair not counted in Pass 2 (due to failing condition 2) may still hash to a frequent bucket in Bitmap 2 by coincidence. The conditions are INDEPENDENT.

Truly frequent pairs always hash to frequent buckets (no matter how many passes).

3. Multihash Algorithm

Idea: Create multiple hash tables in a SINGLE pass (parallel, not sequential).

  • Each table uses different hash function but shares the same pass
  • Pair {i,j} is candidate if: both items frequent AND pair hashes to frequent bucket in ALL hash tables
  • Trade-off: More tables -> more conditions but smaller tables -> higher collision
  • Optimal number of tables: minimize \(p^n\) where p = prob of infrequent pair hashing to frequent bucket
\[P(\text{infrequent pair in } C_2) = p^n\] \[\text{Increasing } n \text{ increases both } n \text{ and } p. \text{ Optimal } n \text{ exists.}\]

4. Limited-Pass Algorithms

Random Sampling

  • Sample fraction p of baskets, run A-Priori in memory with threshold ps
  • False positives: Eliminable with one full pass (verify in full data)
  • False negatives: Hard to eliminate. Can reduce by lowering threshold to 0.9ps

Toivonen's Algorithm

  • Pass 1: Sample, find frequent itemsets (threshold < ps, e.g., 0.9ps). Construct negative border.
  • Negative border: Collection of itemsets NOT frequent in sample but ALL immediate subsets ARE frequent in sample.
  • Pass 2: Check sample-frequent itemsets AND negative border items in full data.
  • If NO border item is frequent in full data -> done (no false positives, no false negatives).
  • If ANY border item is frequent -> algorithm fails, must restart with new sample.
  • No false negatives guarantee: If S is frequent in whole data but not in sample, then either S is in the negative border, or S has an immediate subset T that is frequent in whole but not sample. Recurse until reaching border.
  • Average number of passes is constant, but not deterministic.

SON (Savasere, Omiecinski, Navathe) Algorithm

  • Pass 1: Divide data into 1/p chunks. Run A-Priori on each chunk with threshold ps.
  • Pass 2: Take union of all frequent itemsets from chunks. Verify in full data with threshold s.
  • No false negatives: If itemset has global support >= s, it must have support >= ps in at least one chunk (by pigeonhole principle: if support < ps in every chunk, total support < (1/p) * ps = s, contradiction).
  • No false positives: Verified in Pass 2.

SON with MapReduce

// Pass 1: Find candidate itemsets
map(chunk):                              // each mapper gets one chunk
    run A-Priori on chunk with threshold ps
    for each frequent itemset F:
        emit(F, 1)

reduce(F, counts):                       // collect all candidates
    if F appears at least once:
        output F as candidate

// Pass 2: Find true frequent itemsets
map(candidates, data_portion):           // each mapper gets candidates + data chunk
    for each candidate C:
        count support of C in this portion
        emit(C, local_support)

reduce(C, supports):                     // sum supports across all portions
    if sum(supports) >= s:
        output C as truly frequent

Exam Points

Must-Memorize for L5

  • PCY: Use idle memory in Pass 1 for hash table. Bitmap replaces hash table (1/32 space). Must use triples method.
  • Multistage: Sequential hash tables. Bitmap 1 CANNOT be dropped (conditions are independent).
  • Multihash: Parallel hash tables in single pass. Optimal n minimizes p^n.
  • Random Sampling: FP eliminable (full pass), FN hard to eliminate (reduce threshold)
  • Toivonen: Negative border = not frequent in sample but all immediate subsets are. If border item frequent in full data -> restart.
  • SON: No false negatives (pigeonhole proof). No false positives (Pass 2 verification). Compatible with MapReduce.
  • SON MapReduce pseudocode: Know both passes' map and reduce functions.
T/F Traps:
- "PCY can use triangular matrix" -> FALSE (needs triples for sparsity)
- "Truly frequent pairs always hash to frequent buckets" -> TRUE
- "In Toivonen, checking sample in full data avoids false negatives" -> FALSE (that avoids false POSITIVES; negative border avoids false negatives)
- "Multistage can drop earlier bitmaps" -> FALSE (conditions are independent)
- "SON has false negatives" -> FALSE (guaranteed no FN by pigeonhole)
